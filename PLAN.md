## 二、Yupoo 平台技术分析（已验证）

### 2.1 页面渲染方式

| 项目 | 结论 |
|------|------|
| 渲染方式 | **服务端渲染 (SSR)**，HTML 中直接包含所有数据 |
| 是否需要 JS 引擎 | **不需要**，纯 HTTP 请求即可获取全部内容 |
| CAPTCHA | **无** |
| Cloudflare/WAF | **无** |
| 登录要求 | **无**，公开相册无需认证 |

### 2.2 反爬虫机制

| 机制 | 状态 | 应对方案 |
|------|------|---------|
| **Referer 校验** | ✅ 存在（最关键） | 必须设置 `Referer: https://x.yupoo.com/` 或任意 `*.yupoo.com` 子域名 |
| **频率限制** | ⚠️ 轻微 | 请求间加 0.5~1 秒延迟，并发控制在 5~10 |
| **IP 封禁** | ⚠️ 大量请求时可能触发 | 大规模采集时可考虑代理轮换 |
| **User-Agent 检测** | ❌ 不检测 | 但建议设置正常浏览器 UA 以增加安全性 |
| **Cookie/Token** | ❌ 不需要 | 仅有 `language=zh-CN` cookie，无需特殊处理 |

### 2.3 关键发现：Referer 验证详情

```
无 Referer          → HTTP 567 (拒绝)
Referer: yupoo.com  → HTTP 567 (拒绝) ← 注意：裸域名不行
Referer: *.yupoo.com → HTTP 200 (通过) ← 任意子域名均可
```

**最小可用下载命令：**
```bash
curl -o image.jpeg -e "https://x.yupoo.com/" "https://photo.yupoo.com/{username}/{hash}/big.jpeg"
```

### 2.4 URL 结构

#### 相册列表页
```
https://x.yupoo.com/photos/{username}/albums?page={N}
# 或等效形式
https://{username}.x.yupoo.com/albums?page={N}
```

#### 单个相册页
```
https://x.yupoo.com/photos/{username}/albums/{album_id}
```

#### 图片 CDN 地址
```
https://photo.yupoo.com/{username}/{image_hash}/{size}.{ext}
```

#### 图片尺寸变体

| 尺寸标识 | 大约尺寸 | 文件大小 | 推荐 |
|---------|---------|---------|------|
| `square` | 缩略图正方形裁切 | ~19 KB | ❌ |
| `small` | ~440px 宽 | ~74 KB | ❌ |
| `medium` | ~720px 宽 | ~140 KB | ❌ |
| `big` | ~1080px 宽 | ~211 KB | ✅ **推荐（最大尺寸）** |

> ⚠️ `original`、`large`、`full` 等标识反而返回较小的 ~1003px 图片。`big` 才是最大的渲染尺寸。

### 2.5 HTML 选择器

#### 相册列表页

```html
<a class="album3__main"
   data-album-id="4104348-133516598"
   title="产品名称"
   href="/photos/gyg88/albums/133516598?uid=1&...">
    <img data-src="..." data-origin-src="..." />
    <div class="album3__title">产品名称</div>
</a>
```

- **相册链接**: `a.album3__main` → `href` 属性
- **相册标题（产品名称）**: `a.album3__main` → `title` 属性 或 `div.album3__title` 文本
- **分页总数**: `form.pagination__jumpwrap input[name="page"]` → `max` 属性

#### 单个相册页

```html
<div class="showalbum__children image__main" data-id="254138632">
    <img alt="DSC_0070.JPG"
         data-width="1003"
         data-height="664"
         data-src="https://photo.yupoo.com/gyg88/{hash}/big.jpeg"
         data-origin-src="https://photo.yupoo.com/gyg88/{hash}/{hex}.jpeg"
         data-path="/gyg88/{hash}/{hex}.jpeg"
         data-album-id="133516598"
         src="https://photo.yupoo.com/gyg88/{hash}/small.jpeg">
</div>
```

- **图片容器**: `div.showalbum__children`
- **图片URL (最大)**: `img` → `data-src` 属性（已包含 `big` 尺寸）
- **图片原始URL**: `img` → `data-origin-src` 属性

## 三、采集模块设计方案

### 3.1 技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 语言 | **Python 3.10+** | 生态丰富，requests/aiohttp 成熟 |
| GUI 框架 | **PySide6 (Qt6)** | 功能强大，原生桌面风格，LGPL 许可证可商用，与 PyQt6 API 几乎一致 |
| HTTP 客户端 | **aiohttp**（异步） | 支持并发下载，效率高 |
| HTML 解析 | **BeautifulSoup4 + lxml** | 稳定可靠，CSS 选择器方便 |
| 图片处理(后续) | **Pillow** | 1:1 补齐功能 |
| 异步桥接 | **qasync** | 在 Qt 事件循环中运行 asyncio，避免 GUI 卡死 |

### 3.2 核心模块划分

#### Qt 与 asyncio 集成方案

由于下载器使用 aiohttp（asyncio），而 Qt 有自己的事件循环，需要桥接：

```python
# 方案：使用 qasync 库将 asyncio 集成到 Qt 事件循环
import qasync
from PySide6.QtWidgets import QApplication

app = QApplication(sys.argv)
loop = qasync.QEventLoop(app)
asyncio.set_event_loop(loop)

window = MainWindow()
window.show()

with loop:
    loop.run_forever()
```

下载进度通过 Qt 信号传递到 GUI：

```python
class DownloadWorker(QObject):

    album_done = Signal(str)          # album_name
    error = Signal(str)               # error_message
    finished = Signal()
```

### 3.6 关键设计决策

#### 图片尺寸选择策略
- **默认使用 `big` 尺寸**（~1080px，实测最大）
- 可配置选择 `data-origin-src`（原始文件名路径，约 1003px）
- 提供命令行参数让用户选择尺寸

#### 文件命名策略
```
downloads/
├── [产品名称A]/
│   ├── 01.jpeg
│   ├── 02.jpeg
│   └── 03.jpeg
├── [产品名称B]/
│   ├── 01.jpeg
│   └── 02.jpeg
```
- 文件夹名 = 相册标题（清理非法字符：`\ / : * ? " < > |`）
- 图片按序号命名（01, 02, ...）保持顺序
- 若文件夹名重复，自动追加后缀 `_2`, `_3` ...

#### 断点续传
- 下载前检查本地文件是否已存在
- 已存在则跳过，避免重复下载
- 可选：通过文件大小对比验证完整性

#### 错误处理
- 网络超时：自动重试 3 次，指数退避
- HTTP 567：检查并修正 Referer
- HTTP 404：记录日志，跳过
- 文件写入失败：记录日志，继续下一个

## 四、产品拆分功能设计

### 4.1 问题场景

一个 Yupoo 相册可能包含同款产品的多个配色/款式，例如：

```
相册「AJ1 High」包含 15 张图片:
├── 黑红配色: 正面、侧面、鞋底、细节、上脚 (5张)
├── 黑白配色: 正面、侧面、鞋底、细节 (4张)
└── 全白配色: 正面、侧面、鞋底、细节、鞋盒、上脚 (6张)
```

目标：自动识别并拆分为 3 个独立的产品 SKU 文件夹。

#### 拆分输出目录结构

拆分后的文件夹与原产品文件夹**平级**（同在 `downloads/` 目录下），命名为 `原文件夹名 分组名`：

```
拆分前:
downloads/
└── AJ1 High/
    ├── 01.jpeg ~ 05.jpeg (黑红)
    ├── 06.jpeg ~ 09.jpeg (黑白)
    └── 10.jpeg ~ 15.jpeg (全白)

拆分后:
downloads/
├── AJ1 High 配色_1/    ← 黑红 (5张)
├── AJ1 High 配色_2/    ← 黑白 (4张)
└── AJ1 High 配色_3/    ← 全白 (6张)
```

原产品文件夹在所有文件移走后自动删除。分组名可在 GUI 中自定义编辑。

### 4.2 技术方案：CLIP + 颜色直方图混合聚类

#### 为什么不单独用颜色直方图？
颜色直方图分配色效果好，但两个不同产品恰好颜色相近时会误判。

#### 为什么不单独用 CNN 特征？
CNN 特征理解产品结构，但对同款不同配色的区分不够敏感。

#### 混合方案（推荐）

```
                    图片输入
                   /        \
                  /          \
    ┌────────────────┐  ┌──────────────────┐
    │  CLIP ViT-B/32 │  │  HSV 颜色直方图   │
    │  512 维特征向量  │  │  96 维 (H32+S32+V32) │
    └───────┬────────┘  └────────┬─────────┘
            │                    │
            └──────┬─────────────┘
                   │
            ┌──────▼──────┐
            │  拼接 + L2  │
            │  归一化     │
            │  608 维向量  │
            └──────┬──────┘
                   │
            ┌──────▼──────────────┐
            │  层次聚类            │
            │  AgglomerativeClustering │
            │  distance_threshold │
            └──────┬──────────────┘
                   │
            ┌──────▼──────┐
            │  输出分组    │
            └─────────────┘
```

- **CLIP 特征**负责理解「这是什么产品」（结构、形状、语义）
- **HSV 直方图**负责区分「这是什么颜色」（配色差异）
- 两者拼接后聚类，兼顾结构和颜色

#### 聚类算法：层次聚类 (Agglomerative Clustering)

```python
from sklearn.cluster import AgglomerativeClustering
from sklearn.metrics.pairwise import cosine_distances

distance_matrix = cosine_distances(combined_features)

clustering = AgglomerativeClustering(
    n_clusters=None,             # 不预设分组数，自动判断
    distance_threshold=0.35,     # GUI 滑块可调
    metric='precomputed',
    linkage='average'
)
labels = clustering.fit_predict(distance_matrix)
```

选择层次聚类而非 K-Means / DBSCAN 的原因：
- **K-Means** 需要预先指定 K 值（用户不知道有几个款式）
- **DBSCAN** 的 epsilon 参数难调，且密度不均时效果差（有的款式 2 张图，有的 8 张）
- **层次聚类** 设一个距离阈值即可自动确定分组数，阈值可通过 GUI 滑块微调

## 五、产品查重功能设计

### 5.1 问题场景

用户从多个 Yupoo 店铺、分多批次采集产品，需要检测：
- 不同店铺卖同一款产品（不同角度/背景的照片）
- 同一店铺不同相册重复上架同一产品
- 与历史已下载/已上架的产品重复

### 5.2 为什么 pHash 不够用？

| 场景 | pHash | CLIP |
|------|-------|------|
| 完全相同的图片 | ✅ 能检测 | ✅ 能检测 |
| 同图轻微裁剪/缩放 | ✅ 能检测 | ✅ 能检测 |
| 同产品不同拍摄角度 | ❌ 无法识别 | ✅ 能识别 |
| 同产品不同背景/灯光 | ❌ 无法识别 | ✅ 能识别 |
| 同产品不同店铺的照片 | ❌ 无法识别 | ✅ 能识别 |
| 相似但不同的产品 | ⚠️ 可能误判 | ✅ 能区分 |

pHash 本质是像素级指纹，只要拍摄条件变了就失效。CLIP 理解图片的**语义内容**，知道「这两张图拍的是同一双鞋」。

### 5.3 技术方案：CLIP + HSV 组合特征 + FAISS 向量检索 + SQLite 元数据

> **核心原则：同款不同色 = 不同产品（单 SKU）。**
> 查重使用 CLIP + HSV 组合特征（608 维），与拆分模块共享同一特征提取器，
> HSV 颜色直方图确保不同配色的同款产品不会被误判为重复。

#### 整体架构

```
┌──────────────────────────────────────────────────────────┐
│                    产品特征库                              │
│                                                          │
│  ┌─────────────┐     ┌───────────────────────────────┐   │
│  │ FAISS Index │     │ SQLite 元数据库                │   │
│  │             │     │                               │   │
│  │ 向量 ID → 特│     │ products:                     │   │
│  │ 征向量(608d)│     │   id, name, store, folder,    │   │
│  │             │     │   source_url, download_date,  │   │
│  │ IndexFlatIP │     │   embedding (BLOB备份)         │   │
│  │ (精确余弦)  │     │                               │   │
│  └─────────────┘     │ images:                       │   │
│                      │   id, product_id, file_path,  │   │
│  product_index.faiss │   embedding_id                │   │
│                      └───────────────────────────────┘   │
│                        products.db                       │
└──────────────────────────────────────────────────────────┘
```

- **FAISS IndexFlatIP**：存储 L2 归一化后的 608 维组合向量（CLIP 512d + HSV 96d × 1.5 权重），内积搜索 = 余弦相似度
- **SQLite**：存储产品元数据（名称、来源店铺、文件路径、下载日期等）
- 两者通过 ID 关联

#### 特征提取（共享 CLIP + HSV 模型）

```python
# 产品级组合特征 = 所有图片的 CLIP+HSV 组合特征均值
def compute_product_embedding(image_paths: list[Path]) -> np.ndarray:
    extractor = get_extractor()
    all_feats = []
    for batch in batched(image_paths, DEDUP_BATCH_SIZE):
        feats = extractor.extract_combined_batch(batch)  # (batch, 608)
        all_feats.append(feats)
    all_feats = np.vstack(all_feats)  # (N, 608)
    mean_feat = all_feats.mean(axis=0)
    mean_feat /= np.linalg.norm(mean_feat)  # L2 归一化
    return mean_feat
```

> **关键设计：产品级特征 = 该产品所有图片 CLIP+HSV 组合特征的均值。**
> - CLIP 部分确保不同角度/背景的同一产品能被识别为重复
> - HSV 部分确保同款不同配色被识别为不同产品（单 SKU 需求）

#### 颜色区分效果

| 场景 | 纯 CLIP (旧) | CLIP+HSV (新) |
|------|:-----------:|:------------:|
| 完全重复 | ~0.99 | ~0.99 |
| 同款同色 | ~0.97 | ~0.89 |
| **同款不同色** | **~0.97 误判重复** | **~0.50 正确区分** |

#### FAISS 检索

```python
import faiss

EMBEDDING_DIM = 608  # CLIP(512) + HSV(96)

# 创建索引
base_index = faiss.IndexFlatIP(EMBEDDING_DIM)     # 内积 = 余弦相似度（向量已归一化）
index = faiss.IndexIDMap(base_index)               # 支持自定义 ID

# 添加产品
index.add_with_ids(
    np.array([embedding], dtype=np.float32),
    np.array([product_id], dtype=np.int64)
)

# 查询相似产品
scores, ids = index.search(
    np.array([query_embedding], dtype=np.float32),
    k=5  # 返回 top 5
)

# 持久化
faiss.write_index(index, "data/product_index.faiss")
index = faiss.read_index("data/product_index.faiss")
```

### 5.4 相似度阈值（双层策略）

| 余弦相似度 | 判定 | 动作 |
|-----------|------|------|
| >= 0.92 | 高度重复 | 自动标记为重复，GUI 中高亮显示 |
| 0.85 ~ 0.92 | 疑似重复 | 进入「人工审核队列」 |
| < 0.85 | 不同产品 | 正常入库 |

阈值可在设置中调节。

> **注意：同款不同色的产品（如粉色 vs 黑色足球鞋）因 HSV 颜色特征的加入，
> 相似度会降至 ~0.50，远低于 0.85 阈值，不会被误判为重复。
> 这符合业务需求：每个产品做成单 SKU，不同颜色 = 不同产品。**

| 字段 | 类型 | 选择器 / 定位方式 | 我们的操作 |
|------|------|-----------------|-----------|
| 商品图片 | 上传 | `.upload-img-card.img-upload-icon` → 弹出素材管理弹窗 | 前 5 张 |
| 商品标题 | textarea | `page.locator('textarea').first` | 填写生成标题 |
| 商品类目 | 按钮/弹窗 | `button:has-text("选择商品类目")` | **实测可跳过（非阻断）** |
| 商品型号 | radio | 默认"统一型号" value=0 | 保持默认 |
| 商品价格 | number | `input[placeholder="请填写商品价格"]` | 填写 |
| 商品库存 | number | `input[placeholder="请填写商品库存"]` | 填写 |
| 配送方式 | radio | 默认"快递发货" | 保持默认 |
| 下一步按钮 | button | `button:has-text("下一步")` | 点击进入 Step2 |

#### Step2 商品详情编辑器

| 操作 | 选择器 / 方式 |
|------|-------------|
| 添加图片模块 | `[class*=components__item]:has-text("图片")` → 弹出素材管理弹窗 |
| 创建/提交 | `button:has-text("创建")` |
| 返回上一步 | `button:has-text("上一步")` |

#### 图片上传流程（素材管理弹窗）

```
点击上传区域 / 图片模块
    │
    ▼
弹出「选择图片」弹窗（素材管理库，含历史图片）
    │
    ▼  点击「上传图片」（红色按钮）
弹窗切换到「上传图片」视图
    │
    ▼  定位隐藏的 input[type=file]（display:none, multiple=true）
    │  parent chain: .el-upload.el-upload--picture-card → .avatar-uploader
    │  → .local-img-upload__bottom → .dialog-local-img-upload
    │
    ▼  page.locator('input[type=file]').set_input_files(image_paths)
图片上传到素材库（等待 ~5-8 秒）
    │
    ▼  点击「确定」
弹窗关闭，图片自动应用到对应区域
```

**关键发现：**
- 上传控件不是原生文件选择器，是 Element UI 的 `el-upload` 组件
- 隐藏的 `input[type=file]` 可通过 Playwright 的 `set_input_files()` 直接设置
- 点击确定后图片自动应用，无需手动选择
- 主图限制：max 3MB/张，一次最多 15 张
- 详情图限制：max 5MB/张，一次最多 100 张

#### 完整自动化流程（已验证）

```
Step1: 基本信息
  1. 导航到发布页 URL
  2. 点击 .upload-img-card.img-upload-icon → 弹出素材管理
  3. 点击「上传图片」→ 切换到上传视图
  4. input[type=file].set_input_files(前5张) → 等待上传
  5. 点击「确定」→ 主图设置完成
  6. textarea.first.fill(标题)
  7. input[placeholder=价格].fill(价格)
  8. input[placeholder=库存].fill(库存)
  9. 点击「下一步」→ 进入 Step2

Step2: 商品详情
  10. 关闭「新功能通知」弹窗（如有）→ 点击「我知道了」
  11. 点击「图片」模块 → 弹出素材管理
  12. 点击「上传图片」→ input[type=file].set_input_files(全部图片)
  13. 点击「确定」→ 详情图设置完成
  14. 点击「创建」→ 商品发布完成
```

## 十、风险和注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Yupoo 更改页面结构 | 选择器失效 | 模块化设计，选择器集中配置，便于快速修改 |
| 大量请求被 IP 封禁 | 无法继续下载 | 并发控制 + 请求延迟 + 可选代理支持 |
| 图片 CDN 域名变更 | 下载失败 | CDN 域名配置化 |
| 相册标题含特殊字符 | 文件夹创建失败 | 文件名清理函数 |
| 网络不稳定 | 下载中断 | 断点续传 + 重试机制 |
| CLIP 模型首次下载慢 | 用户体验差 | 显示下载进度条，支持离线预置模型文件 |
| 聚类阈值不准 | 拆分结果差 | GUI 滑块让用户微调 + 手动拖拽修正 |
| FAISS 索引损坏 | 查重数据丢失 | SQLite 中备份 embedding，可重建索引 |
| 微店后台页面改版 | 选择器失效，上架中断 | 选择器集中配置，便于快速修改；失败自动跳过并记录 |
| 浏览器未登录/会话过期 | 操作失败 | 连接前检测登录状态，提示用户重新登录 |
| 上传图片超时 | 单个商品上架失败 | 超时重试，失败记录后可批量重试 |
